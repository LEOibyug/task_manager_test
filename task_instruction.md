# 项目任务书：实验队列与产出管理器 (Exp-Queue-Manager)

## 1. 项目概述

### 1.1 项目目标
构建一个本地桌面应用程序（或Web应用），用于集中管理、监控和操作存储在远程服务器（通过SSH连接）上的机器学习/高性能计算实验。该工具旨在简化多账户、多节点环境下的实验生命周期管理，包括代码版本同步、任务调度、状态监控和结果回收。

### 1.2 核心场景
1.  **主账户**通过Git提交代码更新。
2.  **多个分账户**通过`git pull`同步代码，作为实际执行实验的计算节点账户。
3.  本项目通过SSH连接至**主账户**，读取`experiments/`文件夹下的实验目录。
4.  用户可以：
    *   查看每个实验目录及其内部的`sbatch`脚本。
    *   一键向**指定的分账户**提交`sbatch`任务。
    *   实时监控各账户下正在运行的任务状态、资源占用及输出日志。
    *   一键将**分账户**完成的任务产出（包括`sbatch`输出日志和训练产物）同步回**主账户**的指定目录。

## 2. 核心功能需求

### 2.1 连接与配置管理
*   **服务器连接**：支持配置并保存SSH连接信息（服务器IP， 端口）， 使用本机SSH密钥进行主账户连接验证。
*   **账户管理**：
    *   可配置一个**主账户**的用户名。
    *   可配置多个**分账户**的用户名列表。
    *   为**每个账户**（包括主账户和分账户）配置其代码仓库在服务器上的绝对路径。
*   **配置持久化**：所有连接与账户配置在用户修改后持久化保存（如存储在本地SQLite数据库或JSON文件）， 应用重启后自动加载。

### 2.2 实验列表与状态监控
*   **实验发现**：连接到主账户后， 自动扫描并列出`<主账户仓库路径>/experiments/`目录下的所有子目录（即实验项， 如`exp054_xxxx`）。
*   **脚本识别**：点击某个实验目录， 展示其下的文件列表， 并特别标识出`.sh`和`.sbatch`文件。
*   **全局任务状态面板**：
    *   以表格或看板形式， 实时展示**所有账户**下正在运行（`RUNNING`）、排队（`PENDING`）、完成（`COMPLETED`）或失败（`FAILED`）的任务。
    *   任务信息需包含：任务ID（Job ID）、所属账户、所属实验、状态、开始时间、运行时长、占用的GPU节点（如`gpu1, gpu2, gpu3`）、资源使用率（可选， 需解析`seff`或`sacct`输出）。
    *   实现自动刷新机制（例如每10秒）， 或提供手动刷新按钮。

### 2.3 任务发布
*   **一键发布**：在实验列表或单个实验视图中， 为每个`.sbatch`文件提供“发布到账户X”的按钮。
*   **账户选择**：发布时， 弹出选择框， 允许用户选择**一个分账户**来执行此任务。
*   **自动命令生成**：点击发布后， 项目应通过SSH连接到**所选分账户**， 并执行：
    ```bash
    cd <该分账户的代码仓库路径>
    git pull origin <当前分支>
    sbatch <sbatch文件路径> -w gpu1,gpu2,gpu3
    ```
    *   `-w`参数已预设为`gpu1,gpu2,gpu3`。
*   **并发控制**：在尝试发布新任务前， 先查询该分账户当前已运行和排队的任务数。若已达到**2个**， 则阻止发布并提示用户。
*   **最大时限提示**：发布后， 在任务监控条目上提示其最大允许运行时间为**2天（48小时）**。

### 2.4 输出与日志管理
*   **sbatch输出日志跟踪**：
    *   解析每个已发布任务的`sbatch`脚本， 获取`#SBATCH -o`指定的日志文件路径。
    *   提供“查看日志”按钮， 通过SSH实时读取并展示该日志文件内容。
    *   **大文件处理**：日志文件可能非常大， 读取时采用流式或分页加载， 避免内存溢出。界面提供“跳到末尾”、“搜索关键字”功能。
*   **训练产出目录浏览**：
    *   根据`#SBATCH -J`中的任务名， 推测（或从配置中读取）其对应的`output/<exp_name>/model_config/<tag>/`目录。
    *   提供“浏览产出”按钮， 以树状结构展示该目录下的所有文件和子目录（`ckpt`, `eval`, `tensorboard`, `logs`等）。
    *   支持下载单个文件（如模型权重`*.pth`、`eval`结果）。

### 2.5 输出同步与回收
*   **一键同步**：对于在**分账户**上已完成（状态为`COMPLETED`）的任务， 提供“同步到主账户”按钮。
*   **执行同步**：
    *   通过SSH连接到**主账户**。
    *   执行以下命令序列， 将产出目录和sbatch日志一并同步：
        ```bash
        scp -r <分账户用户名>@<服务器IP>:<分账户仓库路径>/output/<exp_name> <主账户仓库路径>/output/
        scp -r <分账户用户名>@<服务器IP>:<分账户仓库路径>/<sbatch日志文件路径> <主账户仓库路径>/<sbatch日志文件路径>
        ```
    *   *注：* 用户已预先配置好账户间的SSH密钥， 无需处理认证。
*   **状态更新**：同步完成后， 在主界面标记该任务已同步。

## 3. 技术实现要点

### 3.1 架构设计
*   **前端**：推荐使用Electron + Vue/React， 或纯Web应用（Flask/Django + Vue/React）。
*   **后端/核心逻辑**：一个持续运行的服务， 负责与远程服务器交互、管理配置、执行定时任务（如刷新状态）。
*   **通信**：前端与后端通过本地REST API或WebSocket通信， 以实现实时状态推送。

### 3.2 关键技术点
*   **SSH连接**：使用`paramiko`（Python）或`ssh2`（Node.js）库进行SSH连接、命令执行和文件传输。需要维护一个连接池以管理多个会话。
*   **Git版本同步**：在向分账户发布任务前， 自动执行`git pull`， 确保该账户的代码版本与主账户一致。可考虑添加一个“同步所有账户代码”的全局按钮。
*   **并发与异步**：所有SSH操作、任务状态查询都应是异步非阻塞的， 避免界面卡顿。使用线程、协程或任务队列（如Celery）。
*   **并发任务控制**：在发布前， 通过`squeue -u <分账户用户名>`查询任务数。需正确处理`PENDING`和`RUNNING`状态。
*   **配置管理**：使用`configparser`、`JSON`文件或轻量级数据库（如SQLite）存储配置。确保文件权限安全。
*   **前端日志查看**：实现一个日志查看器组件， 支持大文件流式加载、终端样式渲染（ANSI颜色）、关键字高亮和搜索。

## 4. 配置项与持久化方案

| 配置项 | 说明 | 存储位置建议 |
| :--- | :--- | :--- |
| `server_ip` | 服务器IP地址 | 本地配置文件 |
| `server_port` | SSH端口 | 本地配置文件 |
| `main_username` | 主账户用户名 | 本地配置文件 |
| `sub_usernames` | 分账户用户名列表 | 本地配置文件 |
| `repo_paths` | 字典， 映射`{‘username’: ‘/absolute/path/to/repo’}` | 本地配置文件 |
| `refresh_interval` | 状态自动刷新间隔（秒） | 本地配置文件 |

**持久化方案**：在应用用户目录下创建配置文件（如`~/.exp-queue-manager/config.json`）， 应用启动时加载， 修改时实时保存。
